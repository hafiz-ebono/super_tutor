  <behavior>
    clean_extracted_content(text: str, source_type: str = "document") -> str

    Inputs → expected outputs:
    - Text with NFKC ligatures (e.g. "ﬁle") → normalized to "file"
    - Text with trailing spaces per line → trailing spaces stripped
    - Text with 4+ blank lines → collapsed to 2 blank lines
    - Text with HTML tags, source_type="document" → tags stripped
    - Text with HTML tags, source_type="url" → tags preserved (trafilatura produces markdown, not HTML)
    - Empty string → returns empty string
    - Text with fancy quotes (e.g. \u201chello\u201d) → NFKC-normalized

    source_type toggles:
    - "url": skip HTML stripping (preserve markdown headers, bold, links)
    - "document": apply re.sub(r"<[^>]+>", "", text) to strip residual HTML/XML
  </behavior>
  <implementation>
    Create backend/app/extraction/cleaner.py with:
    ```python
    import re
    import unicodedata

    def clean_extracted_content(text: str, source_type: str = "document") -> str:
        text = unicodedata.normalize("NFKC", text)
        text = "\n".join(line.rstrip() for line in text.splitlines())
        text = re.sub(r"\n{3,}", "\n\n", text)
        if source_type == "document":
            text = re.sub(r"<[^>]+>", "", text)
        return text.strip()
    ```
    No external imports needed — stdlib only.
  </implementation>
</feature>

  <behavior>
    extract_document(data: bytes, filename: str) -> str

    PDF cases (filename ends with ".pdf"):
    - text-based PDF with ≥200 chars extracted → returns cleaned, possibly-truncated text
    - scanned/image PDF with <200 chars extracted → raises DocumentExtractionError(error_kind="scanned_pdf")
    - mixed PDF (some scanned pages, total ≥200 chars) → warns via logger but returns text
    - PDF with extracted text > 50,000 chars → truncated at nearest "\n\n" (or ". " fallback), truncation marker appended

    DOCX cases (filename ends with ".docx"):
    - .docx with paragraphs and table cells → returns combined text of all non-empty paragraphs and table cells
    - .docx with text > 50,000 chars → same soft truncation

    Error cases:
    - unsupported file extension → raises DocumentExtractionError(error_kind="unsupported_format")

    Class: DocumentExtractionError(error_kind: str, message: str = "")
    - .error_kind attribute accessible on the exception
    - inherits from Exception

    Never writes to disk — uses BytesIO throughout.

    Truncation constants:
    - SCANNED_PDF_THRESHOLD = 200  (chars across entire document after stripping whitespace)
    - TRUNCATION_LIMIT = 50_000
    - TRUNCATION_MARKER = "\n\n[Content truncated: document exceeds 50,000 characters. Upload a specific chapter or section for full coverage.]"
  </behavior>
  <implementation>
    1. Add to backend/requirements.txt:
       pypdf>=4.0.0
       python-docx>=1.1.0

    2. Create backend/app/extraction/document_extractor.py with the full structure from RESEARCH.md Pattern section.

    Key implementation details:
    - _extract_pdf: use `(page.extract_text() or "")` per page — pypdf returns None for image pages
    - Scanned PDF check: `len(raw.strip()) < SCANNED_PDF_THRESHOLD` — check AFTER joining all pages
    - _extract_docx: iterate doc.paragraphs then doc.tables → rows → cells; check `if para.text.strip()` / `if cell.text.strip()`
    - _soft_truncate: rfind("\n\n", 0, TRUNCATION_LIMIT) first; fall back to rfind(". "); fall back to hard cut at TRUNCATION_LIMIT
    - clean_extracted_content call: import from app.extraction.cleaner; call with source_type="document" AFTER truncation
    - Import structure: use deferred imports (from pypdf / from docx) inside _extract_pdf/_extract_docx — avoids ImportError if lib missing in test env

    - Test cases: scanned_pdf raises, text PDF returns text, docx returns paragraphs+cells, truncation appends marker, unsupported extension raises
